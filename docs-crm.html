Contexto e Objetivo

Por que esse dashboard existe e qual problema ele resolve.

Dor principal hoje Temos muitos ativos de CRM (fluxos de e-mail, fluxos de WhatsApp, campanhas, comunidade) mas não conseguimos saber, com confiança, quais estão puxando a receita e quais precisam de ajuste. As bases de dados estão isoladas e ninguém olha o mesmo número.

O que esse dashboard precisa entregar

  • Local único e confiável de acompanhamento de receita CRM, com leitura diária pela operação e leitura mensal pela gestão.
  • Identificação acionável de quais ativos performam (e quais não) — para decisões de otimização baseadas em dado, não em intuição.
  • Padronização das métricas entre canais (mesmo funil, mesmas fórmulas, mesma atribuição) para que a comparação entre canais faça sentido.

Atribuição

Modelo: Last-Click via UTM A receita de cada canal é atribuída pelo último canal clicado antes da compra. A soma dos 5 canais CRM = Receita CRM Total (sem double-count).

Próximos passos

Sequência sugerida para sair do front-end mockado para produção.

1 · Validar o front-end com o time

Compartilhar este dashboard com gestão e operação. Coletar feedback de UX e métricas antes de gastar engenharia no back-end.

2 · Modelar o schema do Supabase

Criar tabelas espelhando a planilha de mapeamento. Definir granularidade (dia · canal · fluxo · peça) e chaves de UTM.

3 · Conectar fontes via API/integrações

Klaviyo API · Shopify API · Vekta + n8n (já tem pipeline) · Sendflow · Looker (GA4). ETL diário para a base centralizada.

4 · Padronizar UTMs e implementar atribuição last-click

Auditar todos os ativos para garantir UTM padrão. Criar processo para quando aparecer UTM fora do padrão (aba "Sem Atribuição").

5 · Substituir mockados por dados reais

Trocar os mocks do dashboard-crm.html por chamadas ao Supabase. Validar consistência (soma dos 5 canais = Shopify CRM total).

6 · Fase 2 — CAC / LTV / ROAS

Nova página com custos por canal, retorno e visão de margem CRM.
